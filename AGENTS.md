## AI Agents

Guidance for Claude and other AI agents working in this repository.

### Layout & layering

`core` is a modular monolith. Each domain lives under `internal/<module>/` and is split into three layers:

- `api/` — gRPC handlers; translates protobuf requests to/from the service layer.
- `service/` — business logic and validation (e.g. allowed status values).
- `repository/` — data access, using the sqlc-generated queries.

Dependencies flow **api → service → repository** only. Do not import another module's internal packages; share through well-defined boundaries instead. Shared infrastructure (the DB connection pool, generated SQL, logging) lives under `internal/core/`. App config loading lives next to the entrypoint in `cmd/core/config/` (embedded YAML + `CORE_`-prefixed env overrides via viper).

### Database workflow

- Migrations are goose SQL files in `db/core/migrations/`. Create one with `just migrate-gen` (it prints the `goose ... create` command); never hand-rename existing files. Always provide both `+goose Up` and `+goose Down`.
- Hand-written queries live in `db/core/queries/`. After editing a query or migration, run `just generate` to regenerate the sqlc code in `internal/core/db/sqlc/`.
- **Never hand-edit files in `internal/core/db/sqlc/`** — they are generated (`// Code generated by sqlc. DO NOT EDIT.`). If a query is removed, delete its orphaned generated `*.sql.go` file.
- Mirror DB constraints (e.g. `CHECK` value sets) as Go constants in the service layer rather than scattering string literals.

### Commands

Use the `just` recipes rather than ad-hoc commands:

- `just build` — compile (`./cmd/core`).
- `just test` — run tests with the race detector and coverage.
- `just test-integration` — run the `integration`-tagged tests, which spin a real Postgres via testcontainers (requires Docker).
- `just lint` — run `golangci-lint`; keep it at **0 issues** before committing.
- `just generate` — regenerate sqlc code and mocks.
- `just migrate-db` — apply migrations (expects `DATABASE_URL`).

### Transport

The server speaks connectRPC over connect-go (Connect + gRPC + gRPC-Web on one port, cleartext HTTP/2 for local dev). Handlers implement the generated `…connect` handler interfaces from `apis`. gRPC health and reflection are registered alongside each service. Shared logging and panic-recovery interceptors live in `pkg/middleware`. Until auth lands (FNS-96), the caller's user id is read from the temporary `X-User-Id` header.

### Conventions

- Validate and build (`just build` + `just lint`) before committing.
- Follow Conventional Commits (`feat:`, `fix:`, `refactor:`, `chore:`, `docs:`).
- Keep `core` lean: it stores its own ids plus external references and business data only — no auth logic or user profile data (see the docs repo's ADRs).
